# SONGSAI Bank Bridge Stack Decision

## 1. 목적

이 문서는 입금 감지/매칭용 보조 프로젝트를 별도로 만들기 전에,
어떤 기술스택으로 시작할지 먼저 고정하기 위한 결정 문서다.

이 프로젝트의 가칭은 `songsai-bank-bridge` 다.

목표는 아래 세 가지다.

- 입금 알림을 수집한다
- 기존 입금 요청과 자동 매칭한다
- SONGSAI 메인 서비스에서 승인/지급 흐름으로 연결한다

## 2. 왜 별도 프로젝트로 분리하는가

이 기능은 메인 서비스와 성격이 다르다.

- 메인 앱은 사용자 서비스 중심
- bank bridge 는 외부 알림 수신/파싱/매칭 중심

따라서 아래 이유로 분리한다.

- 책임 분리
- 보안 분리
- 추후 교체 용이
- webhook/SMS/email 처리 로직을 메인 앱에서 떼어내기 쉬움

초기에는 같은 서버에 올리되, 코드베이스와 서비스는 분리한다.

## 3. 최종 추천 스택

### 프로젝트 형태

- 별도 저장소
- 별도 앱
- 같은 운영 서버에 별도 포트로 배포

### 언어

- TypeScript

### 런타임

- Node.js

### 프레임워크

- Next.js

### DB

- PostgreSQL

### ORM

- Prisma

### 배포 형태

- 기존 n100 서버에 별도 서비스로 배포
- 예시 포트: `3301`

### 인증/보안

- 외부 사용자 로그인 없음
- 관리자 또는 내부 전용 접근
- 내부 API 는 secret header 기반 인증

## 4. 왜 Next.js 를 선택하는가

이 프로젝트는 webhook 만 받는 아주 작은 서비스로 끝날 수도 있지만,
현실적으로는 운영 화면이 조금 필요할 가능성이 높다.

예:

- 문자 원문 확인
- 파싱 결과 확인
- 자동 매칭 실패 건 확인
- 애매한 매칭 수동 처리
- 최근 알림 로그 확인

Next.js 를 쓰면 아래 장점이 있다.

- 기존 `songsai-music` 과 기술 일관성 유지
- API route 와 운영 UI 를 한 프로젝트 안에 둘 수 있음
- 팀 컨텍스트가 이미 Next.js/TypeScript 중심이라 유지보수가 쉬움

만약 나중에 UI 없는 webhook 서버로 축소하고 싶다면 Express 로 바꿀 수 있지만,
초기 생산성과 운영 편의성 측면에서는 Next.js 가 더 적합하다.

## 5. DB 전략

초기 단계에서는 메인 PostgreSQL 을 같이 사용하는 것을 권장한다.

이유:

- 구현 속도가 빠름
- 운영 복잡도가 낮음
- `DepositRequest` 와 매칭 데이터 연결이 쉬움

주의:

- 프로젝트는 분리하지만 DB 는 공유하는 구조다
- 나중에 필요하면 bank bridge 전용 DB 로 분리할 수 있다

즉 초기 전략은:

- 코드/서비스는 분리
- DB 는 공유

## 6. 서버 배포 전략

초기 배포는 별도 인프라를 늘리지 않고 현재 운영 서버에 같이 올린다.

예상 구조:

- `songsai.org` -> 메인 서비스
- `suno.songsai.org` -> SONGS wrapper
- `bank.songsai.org` 또는 내부 전용 포트 -> bank bridge

추천:

- 외부 공개가 꼭 필요하지 않다면 내부 전용 경로 또는 제한된 서브도메인
- webhook 수신이 필요하므로 최소한 외부에서 접근 가능한 endpoint 는 있어야 함

## 7. 보안 전략

이 프로젝트는 금융성 데이터와 운영 알림을 다루므로,
초기부터 아래 기준을 지킨다.

- webhook endpoint 는 secret signature 또는 bearer secret 검사
- 관리자 화면은 외부 일반 사용자에게 공개하지 않음
- 민감한 알림 원문은 저장하되, 표시 범위는 최소화
- 운영 로그에 계좌번호/토큰/개인정보 전체를 그대로 남기지 않음

## 8. 관찰 대상 입력 채널

초기에는 가장 현실적인 입력 채널부터 시작한다.

우선순위:

1. SMS 포워딩 webhook
2. 이메일 알림 파싱
3. 수동 등록
4. 오픈뱅킹/API 연동

지금 단계에서는 `SMS 포워딩 webhook` 을 1순위로 둔다.

이유:

- 비용이 거의 들지 않음
- 빠르게 테스트 가능
- 운영자가 실제 쓰는 입금 알림 흐름과 가장 가까움

## 9. 초기 기능 범위

MVP 범위는 아래까지만 잡는다.

- 입금 알림 수신
- 원문 저장
- 금액/입금자명 파싱
- 기존 입금 요청과 자동 매칭
- 관리자에게 `자동 매칭 후보` 표시

중요:

- 자동 승인까지 바로 가지 않는다
- 초기엔 `자동 매칭 + 관리자 최종 승인` 방식으로 간다

## 10. 상태 관리 방향

메인 서비스 쪽 입금 요청 상태 예시:

- `PENDING`
- `AUTO_MATCHED`
- `APPROVED`
- `REJECTED`
- `CANCELLED`

bridge 쪽 알림 상태 예시:

- `RECEIVED`
- `PARSED`
- `MATCHED`
- `AMBIGUOUS`
- `UNMATCHED`
- `ERROR`

## 11. 프로젝트 분리 원칙

`songsai-music` 과 `songsai-bank-bridge` 는 아래처럼 나눈다.

### songsai-music 책임

- 사용자 입금 신청 생성
- 관리자 승인/반려
- 유료 크레딧 지급
- 거래내역 기록

### songsai-bank-bridge 책임

- 입금 알림 수신
- 알림 파싱
- 후보 매칭
- 메인 서비스로 매칭 결과 전달

## 12. 지금 시점의 기술 선택 결론

현재 기준 최종 선택은 아래와 같다.

- 프로젝트명: `songsai-bank-bridge`
- 프레임워크: `Next.js`
- 언어: `TypeScript`
- 런타임: `Node.js`
- DB: `PostgreSQL` 공유
- ORM: `Prisma`
- 배포: 기존 n100 서버 별도 포트
- 입력 채널: SMS 포워딩 webhook 우선
- 운영 방식: 자동 승인 X, 자동 매칭 + 관리자 승인 O
