# Media Storage And Streaming Plan

이 문서는 SONGSAI의 음원/비디오 저장, 다운로드, 스트리밍 구성을 정리한 설계 문서다.

대상 장비:

- 신규 서버
  - CPU: Xeon E5-2650 v3
  - GPU: RTX 3070 Ti
  - RAM: 64GB
  - 저장장치: 1TB SSD + 500GB 7200RPM HDD
- 구형 NAS
  - WD My Cloud

## 결론 요약

1. 앱, DB, wrapper, 최신 미디어는 신규 서버 SSD에 둔다.
2. 오래된 미디어와 로컬 백업은 신규 서버 HDD로 내린다.
3. 장기 보관과 추가 백업은 WD My Cloud로 보낸다.
4. 스트리밍 서버가 필요하면 WD My Cloud에 직접 두기보다 신규 서버에 두고, NAS는 저장소로 마운트해 쓰는 방식이 좋다.
5. SONGSAI 앱 안에서 단순 다운로드만 제공할 때는 별도 미디어 서버보다 정적 파일 서빙이 더 단순하고 효율적이다.

## 저장 계층 설계

### 1. Hot Storage: 1TB SSD

여기는 실시간 서비스와 자주 접근하는 데이터를 둔다.

- 운영체제
- Docker / 앱 실행 파일
- `D:\music`
- `D:\songsai-bank-bridge`
- `D:\wrapper`
- PostgreSQL 데이터
- wrapper 브라우저 프로필 / 세션 파일
- 최근 생성된 mp3 / 이미지 / mp4
- 비디오 렌더 임시 파일

이유:

- DB와 앱 반응성
- wrapper 세션 안정성
- ffmpeg / 브라우저 렌더 I/O
- 최근 미디어 재생 / 다운로드 성능

### 2. Warm Storage: 500GB HDD

여기는 속도보다 용량 확보가 중요한 파일을 둔다.

- 오래된 생성 mp3
- 오래된 생성 mp4
- 오래된 비디오 보조 자산
- DB dump
- 로그 아카이브
- 배포 백업 zip

권장 규칙:

- 최근 7일~30일 파일만 SSD 유지
- 그보다 오래된 파일은 HDD로 이동

### 3. Cold Storage: WD My Cloud

여기는 장기 보관과 백업용이다.

- 오래된 mp3 / mp4 장기 보관
- PostgreSQL dump 백업
- 코드 스냅샷 / 문서 백업
- 장애 복구용 아카이브

## NAS의 역할 판단

WD My Cloud 같은 구형 NAS는 계산 서버로는 부적합하지만, 정적 파일 저장소로는 충분히 쓸 수 있다.

특히 SONGSAI 파일 크기 가정:

- mp3 약 3MB
- mp4 약 8MB

이 정도면 파일 저장, 다운로드, 저부하 스트리밍 자체는 가능하다.

하지만 아래 용도는 권장하지 않는다.

- DB 실행
- Next.js 앱 실행
- wrapper 브라우저 자동화
- ffmpeg 렌더
- 트래픽이 몰리는 핵심 실시간 서비스

## 다운로드와 스트리밍 방식 선택

### A. 가장 단순한 방식

SONGSAI 앱이 파일 경로를 알고 있고, 사용자가 다운로드 또는 재생 버튼을 눌렀을 때 정적 파일 URL을 내려주는 방식

추천 상황:

- SONGSAI 안에서만 파일을 사용
- 별도 미디어 앱 UI가 꼭 필요하지 않음
- 구현을 가장 단순하게 하고 싶음

장점:

- 구조가 단순함
- 권한 통제를 앱에서 직접 하기 쉬움
- 메타데이터를 앱 DB와 그대로 연결 가능

추천 구성:

- 최신 파일: SSD에서 Nginx/Caddy 직접 서빙
- 오래된 파일: HDD 또는 NAS 경로를 read-only 마운트해서 서빙
- `Range` 요청 지원으로 오디오/비디오 seek 가능하게 설정

### B. Jellyfin 구축

추천도: 높음

추천 상황:

- 음원/비디오 라이브러리처럼 탐색하고 싶음
- TV/모바일 앱 클라이언트도 쓰고 싶음
- Plex Pass 비용 없이 가고 싶음

장점:

- 무료 오픈소스
- 음악 비디오 라이브러리 지원
- GPU 하드웨어 가속 지원
- 브라우저 / 모바일 / TV 클라이언트 생태계가 괜찮음

추천 배치:

- Jellyfin 서버는 신규 서버 SSD에서 실행
- 라이브러리는 SSD + HDD + NAS 마운트 경로를 함께 읽게 구성
- NAS는 저장소 역할만 하고, Jellyfin 프로세스는 NAS에 올리지 않음

### C. Plex 구축

추천도: 조건부

추천 상황:

- 더 polished한 상용 느낌의 클라이언트 경험을 원함
- Plex 생태계를 선호함
- 하드웨어 트랜스코딩에 Plex Pass 비용을 감수할 수 있음

장점:

- 앱/클라이언트 경험이 좋음
- 설정이 비교적 쉬움

단점:

- 하드웨어 가속은 Plex Pass 필요
- 비용/제약이 생김

## 내 추천

### 1순위: SONGSAI 기본 다운로드/재생은 앱에서 직접 제공

가장 먼저는 미디어 서버를 크게 벌리지 말고, `music` 앱 안에서 정적 파일 다운로드/재생 경로를 정리하는 것이 좋다.

이유:

- 권한 처리와 서비스 흐름을 한 곳에서 제어 가능
- 기존 `MusicAsset` 구조와 잘 맞음
- 구현 복잡도가 낮음

### 2순위: 별도 라이브러리형 스트리밍이 필요하면 Jellyfin

만약 “음악/비디오를 넷플릭스나 개인 미디어 라이브러리처럼 보고 싶다”가 목표라면 Plex보다 Jellyfin이 더 잘 맞는다.

이유:

- 비용 부담이 적음
- GPU 가속 활용 가능
- self-hosted 친화적
- 구형 NAS는 저장소로만 써도 됨

### 3순위: Plex는 상용 클라이언트 경험이 꼭 필요할 때만

Plex 자체는 나쁘지 않지만, SONGSAI 현재 단계에서는 비용과 구조 복잡도를 늘릴 가능성이 있다.

## 권장 운영 구조

### Phase 1

가장 먼저 추천하는 실전 구조

- 앱/DB/wrapper: SSD
- 최신 미디어: SSD
- 오래된 미디어: HDD
- 장기 백업: NAS
- 앱 다운로드/재생: Nginx/Caddy 또는 앱 API 기반 정적 파일 서빙

### Phase 2

라이브러리형 사용성이 필요해질 때

- Jellyfin를 신규 서버에 설치
- 라이브러리 경로:
  - SSD의 최신 미디어 폴더
  - HDD의 아카이브 폴더
  - NAS의 장기 저장 폴더(읽기 전용)

### Phase 3

사용량이 늘어나면

- SSD에는 최신/인기 파일만 유지
- 오래된 파일은 HDD/NAS 중심
- 자주 재생되는 파일만 캐시
- 스트리밍과 다운로드 경로를 분리

## GPU 활용 전략

RTX 3070 Ti 는 스트리밍 서버를 신규 서버에 두는 쪽에서 가치가 크다.

추천 활용:

- Jellyfin 하드웨어 트랜스코딩
- ffmpeg 하드웨어 인코딩
- 향후 보컬 분리 / AI 처리

중요:

- 가능하면 Direct Play 위주로 설계하고, 트랜스코딩은 보조 수단으로 본다.
- mp3 3MB, mp4 8MB 수준이면 트랜스코딩 없이 직접 재생 가능한 경우가 많다.

## 최종 판단

### "WD My Cloud로 스트리밍 서버까지 직접 운영할까?"

비추천.

이유:

- 앱/트랜스코딩/메타데이터 처리 성능이 부족할 가능성이 큼
- 장애 시 병목이 NAS에 몰림

### "WD My Cloud에 파일을 두고 다운로드/스트리밍 원본 저장소로 쓸까?"

추천 가능.

이유:

- 파일 크기가 크지 않음
- 정적 파일 저장소 역할에는 적합함

### "Plex vs Jellyfin?"

현재 기준 추천은 Jellyfin.

- 이유: 비용, 자유도, self-hosted 친화성, GPU 활용

4. 필요하면 Jellyfin를 신규 서버에만 올리고 NAS는 마운트 저장소로 사용
